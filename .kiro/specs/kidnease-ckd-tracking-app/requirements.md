# Requirements Document: Kidnease CKD Tracking Application

## Introduction

Kidnease is a comprehensive mobile health (mHealth) application designed to help users prevent and manage Chronic Kidney Disease (CKD) through AI-powered dietary tracking and risk assessment. The system uses multimodal AI vision to analyze food images captured by users and provides real-time nutritional risk assessments against KDIGO (Kidney Disease: Improving Global Outcomes) clinical guidelines.

The application operates on a privacy-first, cost-effective "on-the-fly" processing model where raw food images are captured, processed through AI vision services, and discarded after analysis. Only text-based assessment history and personalized dietary limits are persisted to cloud storage.

The target platform is Android smartphones using Flutter framework, with Firebase backend services and integration with Google Gemini 3.0 Pro API for multimodal AI vision and FatSecret API for commercial nutritional data.

## Glossary

- **Kidnease_App**: The Flutter mobile application running on Android devices
- **Image_Capture_Module**: The camera interface component for capturing food images
- **Cloud_Storage_Service**: Firebase Cloud Storage service for temporary image blob storage
- **Gemini_API_Client**: Service class handling integration with Google Gemini 3.0 Pro API
- **FatSecret_API_Client**: Service class handling integration with FatSecret nutritional database API
- **Risk_Assessment_Engine**: Backend logic component that orchestrates AI vision analysis and risk evaluation
- **Firestore_Database**: Cloud Firestore NoSQL database for persistent storage
- **Authentication_Service**: Firebase Authentication service for user identity management
- **Dashboard_Module**: UI component displaying longitudinal dietary analytics and progress visualization
- **Notification_Service**: System component for displaying real-time risk alerts to users
- **Dietary_Profile**: User-specific KDIGO-compliant daily nutritional limits (sodium, potassium, phosphorus, protein)
- **KDIGO_Guidelines**: Kidney Disease: Improving Global Outcomes clinical practice guidelines for CKD dietary management
- **Risk_Level**: Classification of dietary risk as either "High Risk" or "Safe" based on KDIGO limits
- **Assessment_Record**: Persistent JSON document containing risk analysis results stored in Firestore
- **Image_URL**: Secure temporary URL generated by Firebase Cloud Storage for image blob access
- **Multimodal_AI_Vision**: Google Gemini's capability to process both image and text inputs for object detection and OCR
- **Filipino_Dietary_Alternative**: Culturally relevant renal-friendly food substitution suggestions

## Requirements

### Requirement 1: User Authentication and Profile Management

**User Story:** As a CKD patient or at-risk individual, I want to securely register and authenticate with the application, so that my health data remains private and personalized to my condition.

#### Acceptance Criteria

1. THE Authentication_Service SHALL create a new user account with email and password credentials
2. WHEN a user submits valid login credentials, THE Authentication_Service SHALL authenticate the user and grant access to the application
3. WHEN a user submits invalid login credentials, THE Authentication_Service SHALL reject authentication and return a descriptive error message
4. THE Firestore_Database SHALL store user profile data including userID, userRole, name, email, and createdAt timestamp
5. WHEN a user successfully authenticates, THE Kidnease_App SHALL retrieve the associated Dietary_Profile from Firestore_Database
6. THE Authentication_Service SHALL maintain secure session state throughout the user's interaction with the application

### Requirement 2: Personalized Dietary Profile Configuration

**User Story:** As a healthcare provider or informed patient, I want to configure personalized KDIGO-compliant daily nutritional limits, so that risk assessments are tailored to my specific CKD stage and medical needs.

#### Acceptance Criteria

1. THE Kidnease_App SHALL provide a user interface for entering daily sodium limit in milligrams
2. THE Kidnease_App SHALL provide a user interface for entering daily potassium limit in milligrams
3. THE Kidnease_App SHALL provide a user interface for entering daily phosphorus limit in milligrams
4. THE Kidnease_App SHALL provide a user interface for entering daily protein limit in grams
5. WHEN a user saves dietary limits, THE Firestore_Database SHALL persist a Dietary_Profile document containing profileID, userID, dailySodiumLimit, dailyPotassiumLimit, dailyPhosphorusLimit, dailyProteinLimit, and lastUpdated timestamp
6. WHEN dietary limits are updated, THE Firestore_Database SHALL update the lastUpdated timestamp to the current time
7. THE Kidnease_App SHALL validate that all nutritional limit values are positive numbers greater than zero

### Requirement 3: Food Image Capture and Upload

**User Story:** As a user, I want to capture images of food labels or plated meals using my smartphone camera, so that the system can analyze nutritional content without manual data entry.

#### Acceptance Criteria

1. THE Image_Capture_Module SHALL provide a camera interface for capturing JPEG images of food items
2. WHEN a user captures an image, THE Image_Capture_Module SHALL compress the image to reduce file size while maintaining OCR readability
3. WHEN an image is captured, THE Cloud_Storage_Service SHALL upload the image blob to Firebase Cloud Storage
4. WHEN an image upload completes successfully, THE Cloud_Storage_Service SHALL generate and return a secure Image_URL with temporary access permissions
5. IF an image upload fails, THEN THE Kidnease_App SHALL display an error message and allow the user to retry
6. THE Image_Capture_Module SHALL support capturing images of both physical food labels and un-barcoded plated Filipino meals

### Requirement 4: Commercial Nutritional Data Retrieval

**User Story:** As a user scanning packaged food products, I want the system to retrieve baseline commercial nutritional data, so that AI analysis has accurate reference information for branded items.

#### Acceptance Criteria

1. WHERE a captured food image contains identifiable commercial product information, THE FatSecret_API_Client SHALL query the FatSecret API for nutritional data
2. WHEN the FatSecret API returns nutritional data, THE FatSecret_API_Client SHALL extract sodium, potassium, phosphorus, and protein values
3. IF the FatSecret API does not return matching nutritional data, THEN THE Risk_Assessment_Engine SHALL proceed with AI vision analysis using only image-based extraction
4. THE FatSecret_API_Client SHALL handle API rate limits and return appropriate error messages when limits are exceeded
5. THE FatSecret_API_Client SHALL complete API queries within 5 seconds or return a timeout error

### Requirement 5: AI-Powered Nutritional Information Extraction

**User Story:** As a user, I want the system to automatically extract nutritional information from food images using AI vision, so that I don't have to manually transcribe nutrition labels or estimate meal content.

#### Acceptance Criteria

1. WHEN an Image_URL is available, THE Gemini_API_Client SHALL send the Image_URL to the Google Gemini 3.0 Pro API with a multimodal prompt
2. THE Gemini_API_Client SHALL include the user's Dietary_Profile limits in the API request payload
3. WHERE FatSecret nutritional data is available, THE Gemini_API_Client SHALL include this data in the API request as reference information
4. THE Gemini_API_Client SHALL request object detection and OCR analysis from the Multimodal_AI_Vision service
5. WHEN Gemini API returns results, THE Gemini_API_Client SHALL parse the JSON response containing detected food name, sodium, potassium, phosphorus, and protein values
6. IF the Gemini API returns an error or invalid response, THEN THE Gemini_API_Client SHALL return a descriptive error message to the user
7. THE Gemini_API_Client SHALL complete API requests within 10 seconds or return a timeout error

### Requirement 6: Dietary Risk Assessment and Classification

**User Story:** As a CKD patient, I want the system to evaluate whether detected nutritional content exceeds my KDIGO limits, so that I can make informed decisions about food consumption.

#### Acceptance Criteria

1. WHEN nutritional values are extracted, THE Risk_Assessment_Engine SHALL compare sodium value against the user's dailySodiumLimit
2. WHEN nutritional values are extracted, THE Risk_Assessment_Engine SHALL compare potassium value against the user's dailyPotassiumLimit
3. WHEN nutritional values are extracted, THE Risk_Assessment_Engine SHALL compare phosphorus value against the user's dailyPhosphorusLimit
4. WHEN nutritional values are extracted, THE Risk_Assessment_Engine SHALL compare protein value against the user's dailyProteinLimit
5. IF any nutritional value exceeds its corresponding daily limit, THEN THE Risk_Assessment_Engine SHALL classify the Risk_Level as "High Risk"
6. IF all nutritional values are within their corresponding daily limits, THEN THE Risk_Assessment_Engine SHALL classify the Risk_Level as "Safe"
7. THE Risk_Assessment_Engine SHALL generate a cause-and-effect explanation describing which specific nutrients exceed limits and the resulting renal strain

### Requirement 7: Culturally Relevant Dietary Alternative Recommendations

**User Story:** As a Filipino user, I want to receive renal-friendly food substitution suggestions that align with my cultural dietary preferences, so that I can maintain my lifestyle while protecting my kidney health.

#### Acceptance Criteria

1. WHEN a Risk_Level is classified as "High Risk", THE Gemini_API_Client SHALL request Filipino_Dietary_Alternative suggestions from the Gemini API
2. THE Gemini_API_Client SHALL provide context about Filipino cuisine and cultural food preferences in the API prompt
3. WHEN Gemini API returns alternatives, THE Kidnease_App SHALL display at least one renal-friendly food substitution suggestion
4. THE Filipino_Dietary_Alternative suggestions SHALL maintain similar taste profiles and culinary functions to the original food item
5. THE Filipino_Dietary_Alternative suggestions SHALL have lower sodium, potassium, phosphorus, or protein content than the original food item based on which nutrient exceeded limits

### Requirement 8: Real-Time Risk Notification Display

**User Story:** As a user, I want to receive immediate visual feedback about dietary risk after capturing a food image, so that I can decide whether to consume the food before eating.

#### Acceptance Criteria

1. WHEN a Risk_Level is determined, THE Notification_Service SHALL display a real-time UI notification within 2 seconds
2. WHERE the Risk_Level is "High Risk", THE Notification_Service SHALL display a prominent warning notification with red visual indicators
3. WHERE the Risk_Level is "Safe", THE Notification_Service SHALL display a confirmation notification with green visual indicators
4. THE Notification_Service SHALL include the cause-and-effect explanation text in the notification display
5. WHERE Filipino_Dietary_Alternative suggestions are available, THE Notification_Service SHALL display these alternatives in the notification
6. THE Notification_Service SHALL allow users to dismiss notifications after reviewing the information

### Requirement 9: Assessment History Persistence

**User Story:** As a user, I want the system to save my dietary assessment history, so that I can review past food choices and track my dietary patterns over time.

#### Acceptance Criteria

1. WHEN a risk assessment completes, THE Firestore_Database SHALL create a DietaryAssessments document containing assessmentID, userID, timestamp, imageURL, detectedFoodName, riskLevel, and explanationText
2. WHEN a risk assessment completes, THE Firestore_Database SHALL create an ExtractedNutrients document containing nutrientRecordID, assessmentID, sodiumValue, potassiumValue, phosphorusValue, and proteinValue
3. WHEN a risk notification is displayed, THE Firestore_Database SHALL create a RiskNotifications document containing notificationID, userID, assessmentID, severityLevel, displayMessage, and timestamp
4. THE Firestore_Database SHALL index DietaryAssessments documents by userID and timestamp for efficient querying
5. THE Firestore_Database SHALL maintain referential integrity between DietaryAssessments and ExtractedNutrients documents using assessmentID
6. THE Kidnease_App SHALL retrieve assessment history ordered by timestamp in descending order

### Requirement 10: Privacy-First Image Lifecycle Management

**User Story:** As a privacy-conscious user, I want my food images to be automatically deleted after processing, so that my visual dietary data is not stored long-term in the cloud.

#### Acceptance Criteria

1. WHEN a risk assessment completes successfully, THE Cloud_Storage_Service SHALL delete the image blob from Firebase Cloud Storage
2. WHEN a risk assessment fails or times out, THE Cloud_Storage_Service SHALL delete the image blob from Firebase Cloud Storage
3. THE Cloud_Storage_Service SHALL set automatic deletion policies for image blobs older than 24 hours as a fallback mechanism
4. THE Firestore_Database SHALL store only the Image_URL reference in DietaryAssessments documents, not the actual image data
5. WHEN an image is deleted, THE Image_URL SHALL become invalid and return access denied errors if accessed
6. THE Kidnease_App SHALL not cache or store raw image data locally on the device after upload completes

### Requirement 11: Longitudinal Dietary Analytics Dashboard

**User Story:** As a user, I want to view visual analytics of my daily nutritional intake over time, so that I can understand my dietary patterns and progress toward kidney health goals.

#### Acceptance Criteria

1. THE Dashboard_Module SHALL display a chart showing daily sodium consumption versus dailySodiumLimit over the past 30 days
2. THE Dashboard_Module SHALL display a chart showing daily potassium consumption versus dailyPotassiumLimit over the past 30 days
3. THE Dashboard_Module SHALL display a chart showing daily phosphorus consumption versus dailyPhosphorusLimit over the past 30 days
4. THE Dashboard_Module SHALL display a chart showing daily protein consumption versus dailyProteinLimit over the past 30 days
5. THE Dashboard_Module SHALL display progress rings indicating current day's consumption as a percentage of daily limits
6. WHEN a user selects a specific date on the chart, THE Dashboard_Module SHALL display detailed assessment records for that date
7. THE Dashboard_Module SHALL aggregate ExtractedNutrients data by date to calculate daily totals
8. THE Dashboard_Module SHALL update visualizations in real-time when new assessment records are created

### Requirement 12: Gemini API Response Parsing and Validation

**User Story:** As a system administrator, I want the application to robustly parse and validate AI API responses, so that malformed or unexpected responses do not crash the application or provide incorrect risk assessments.

#### Acceptance Criteria

1. WHEN Gemini API returns a response, THE Gemini_API_Client SHALL validate that the response contains a valid JSON structure
2. THE Gemini_API_Client SHALL validate that the JSON response contains required fields: riskLevel, explanationText, detectedFoodName, and nutritional values
3. IF the JSON response is missing required fields, THEN THE Gemini_API_Client SHALL return a validation error and request user to retry
4. THE Gemini_API_Client SHALL validate that riskLevel value is either "High Risk" or "Safe"
5. THE Gemini_API_Client SHALL validate that nutritional values are non-negative numbers
6. IF nutritional values are missing or invalid, THEN THE Gemini_API_Client SHALL use zero as the default value and log a warning
7. THE Gemini_API_Client SHALL sanitize explanationText to remove potentially harmful content before display

### Requirement 13: Offline Mode and Network Error Handling

**User Story:** As a user in areas with unreliable internet connectivity, I want the application to gracefully handle network failures and provide clear feedback, so that I understand when features are unavailable.

#### Acceptance Criteria

1. WHEN the device has no internet connectivity, THE Kidnease_App SHALL display a network unavailable message
2. WHEN the device has no internet connectivity, THE Kidnease_App SHALL disable the Image_Capture_Module to prevent failed uploads
3. WHILE the device has no internet connectivity, THE Dashboard_Module SHALL display cached assessment history from local storage
4. IF an API request fails due to network timeout, THEN THE Kidnease_App SHALL display a retry option to the user
5. THE Kidnease_App SHALL cache the most recent 100 assessment records locally for offline viewing
6. WHEN internet connectivity is restored, THE Kidnease_App SHALL synchronize any pending data with Firestore_Database

### Requirement 14: Cost-Effective Cloud Resource Usage

**User Story:** As a system operator, I want the application to minimize cloud storage and API costs, so that the service remains financially sustainable at scale.

#### Acceptance Criteria

1. THE Image_Capture_Module SHALL compress images to a maximum file size of 2 MB before upload
2. THE Cloud_Storage_Service SHALL use Firebase Cloud Storage lifecycle policies to automatically delete images older than 24 hours
3. THE Gemini_API_Client SHALL implement request batching where multiple assessments can be processed in a single API call when technically feasible
4. THE Firestore_Database SHALL use indexed queries to minimize read operations
5. THE Kidnease_App SHALL implement local caching to reduce redundant Firestore queries for static data like Dietary_Profile
6. THE Kidnease_App SHALL limit dashboard analytics queries to the most recent 90 days of data by default

### Requirement 15: KDIGO Guideline Compliance Validation

**User Story:** As a healthcare provider, I want the system to enforce KDIGO-compliant dietary limits, so that risk assessments align with evidence-based clinical practice guidelines for CKD management.

#### Acceptance Criteria

1. THE Kidnease_App SHALL provide reference documentation of KDIGO-recommended daily limits for CKD stages 1-5
2. WHEN a user configures a Dietary_Profile, THE Kidnease_App SHALL display KDIGO-recommended ranges for each nutrient based on CKD stage
3. THE Kidnease_App SHALL allow users to set custom limits within 50% to 200% of KDIGO recommendations
4. IF a user attempts to set limits outside the acceptable range, THEN THE Kidnease_App SHALL display a warning message
5. THE Risk_Assessment_Engine SHALL use only the user's configured Dietary_Profile limits for risk classification, not hardcoded values
6. THE Kidnease_App SHALL include a disclaimer that dietary recommendations should be validated with healthcare providers

### Requirement 16: Multimodal AI Vision Prompt Engineering

**User Story:** As a system designer, I want the Gemini API prompts to be optimized for accurate nutritional extraction and culturally relevant recommendations, so that AI responses meet user needs consistently.

#### Acceptance Criteria

1. THE Gemini_API_Client SHALL construct prompts that explicitly request object detection and OCR for nutritional labels
2. THE Gemini_API_Client SHALL construct prompts that request simplified cause-and-effect explanations of renal strain suitable for non-medical users
3. THE Gemini_API_Client SHALL construct prompts that request Filipino_Dietary_Alternative suggestions with cultural context
4. THE Gemini_API_Client SHALL include the user's Dietary_Profile limits in the prompt to enable personalized risk assessment
5. THE Gemini_API_Client SHALL request JSON-formatted responses with a specified schema to ensure consistent parsing
6. THE Gemini_API_Client SHALL include examples of expected response format in the prompt to improve output quality

### Requirement 17: State Management and UI Responsiveness

**User Story:** As a user, I want the application interface to remain responsive during image processing and API calls, so that I can continue using other features while assessments are in progress.

#### Acceptance Criteria

1. WHEN an image upload or API call is in progress, THE Kidnease_App SHALL display a loading indicator
2. WHILE an assessment is processing, THE Kidnease_App SHALL allow users to navigate to the Dashboard_Module or profile settings
3. THE Kidnease_App SHALL use asynchronous programming patterns to prevent UI blocking during network operations
4. WHEN an assessment completes, THE Kidnease_App SHALL update the UI state and display results without requiring manual refresh
5. THE Kidnease_App SHALL use Provider or Riverpod state management to maintain consistent state across UI components
6. THE Kidnease_App SHALL handle rapid successive image captures by queuing assessments and processing them sequentially

### Requirement 18: Error Logging and Diagnostics

**User Story:** As a system administrator, I want comprehensive error logging for API failures and processing errors, so that I can diagnose and resolve issues affecting user experience.

#### Acceptance Criteria

1. WHEN an API call fails, THE Kidnease_App SHALL log the error message, timestamp, userID, and request parameters
2. WHEN image processing fails, THE Kidnease_App SHALL log the error type, Image_URL, and failure reason
3. THE Kidnease_App SHALL log all Gemini API response validation failures with the raw response payload
4. THE Kidnease_App SHALL use Firebase Crashlytics for crash reporting and error tracking
5. THE Kidnease_App SHALL implement log levels (INFO, WARNING, ERROR) to categorize diagnostic information
6. THE Kidnease_App SHALL not log sensitive user data such as email addresses or health information in plain text

### Requirement 19: Assessment Record Search and Filtering

**User Story:** As a user, I want to search and filter my assessment history by food name, date range, or risk level, so that I can quickly find specific dietary records.

#### Acceptance Criteria

1. THE Dashboard_Module SHALL provide a search input field for filtering assessments by detectedFoodName
2. THE Dashboard_Module SHALL provide date range filters for selecting assessments within a specific time period
3. THE Dashboard_Module SHALL provide a filter toggle for displaying only "High Risk" assessments
4. WHEN a user applies filters, THE Dashboard_Module SHALL query Firestore_Database with appropriate filter conditions
5. THE Dashboard_Module SHALL display filtered results in descending chronological order
6. THE Dashboard_Module SHALL display a count of total assessments matching the current filter criteria

### Requirement 20: User Onboarding and Tutorial Flow

**User Story:** As a first-time user, I want guided onboarding that explains how to use the application and configure my dietary profile, so that I can start tracking my diet effectively.

#### Acceptance Criteria

1. WHEN a user first launches the application after registration, THE Kidnease_App SHALL display an onboarding tutorial
2. THE Kidnease_App SHALL guide users through configuring their initial Dietary_Profile during onboarding
3. THE Kidnease_App SHALL demonstrate the image capture workflow with visual examples during onboarding
4. THE Kidnease_App SHALL explain the meaning of Risk_Level classifications and notification colors during onboarding
5. THE Kidnease_App SHALL allow users to skip onboarding and access it later from settings
6. THE Kidnease_App SHALL mark onboarding as completed in the user's profile to prevent repeated displays
